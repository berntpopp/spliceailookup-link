# spliceailookup-link — Beyond 9.5 (Design)

**Date:** 2026-06-12
**Author:** MCP engineering (driven by a live self-test of the deployed server)
**Status:** Design — pending implementation
**Baseline:** v0.3.0 corrective pass (branch `eval-improvements-2`) → target **v0.4.0**

## 1. Why

After the v0.3.0 corrective pass (F6–F10 + #2/#4/#5), a live self-test was run as
a critical LLM-consumer on a real cross-server research workflow:

> get domains for PNKP using uniprot-link → identify a PNKP coding variant →
> ask spliceailookup-link what it does to splicing.

The self-test confirmed the v0.3.0 targets are real (the deployed v0.2.0
`predict_splicing("1-169549811-C-A")` returned `agreement.verdict:"discordant"`
while its `headline` said "models agree" — the live F6 bug v0.3.0 fixes), and it
exposed two composability/decision-completeness gaps that v0.3.0 does **not**
address. Closing them pushes the cross-server and schema dimensions toward 10.

### Self-test evidence

`predict_splicing("NM_007254.4:c.1123G>T", response_mode="full")` (PNKP):

- Resolved cleanly to `19-49862188-C-A`, both models scored, clean "no splice
  effect" answer (`agreement: concordant_low`), `_meta.resolved_consequence:
  "missense_variant"`.
- **Gap G1:** `_meta.see_also` listed gnomad-link / genereviews-link / gtex-link
  but **not uniprot-link** — even though the workflow originated in uniprot and
  the variant is a coding (missense) change whose obvious next question is "which
  protein domain/feature does residue X sit in?". The cross-server loop back to
  protein context is missing.
- **Gap G2:** the molecular consequence (`missense_variant`) was present only in
  `_meta.resolved_consequence`. The top-level result and the headline never state
  the variant *type*, so a consumer asking "what does this variant do?" must read
  both the splice verdict and dig into `_meta` to assemble the full picture.

## 2. Goal & score model

Lift the dimensions still short of 10 after v0.3.0, without new latency or a new
upstream dependency:

| Axis | Dimension | v0.3.0 | Target | Lever |
|---|---|---|---|---|
| Consumer | Composability / chaining | 9.5 | 10 | **G1** uniprot-link in see_also (4th sibling, closes the protein-context loop) |
| Consumer | Schema / decision-completeness | 9.5 | 10 | **G2** top-level `molecular_consequence` + headline mention |
| Consumer | Discoverability | 9.5 | 9.5+ | capabilities documents the uniprot delegation + the new field |

Latency stays the upstream-bound ceiling (the background-task pattern from #2 is
the mitigation; no further server lever exists). Both changes are additive,
token-cheap, and respect the "delegate cross-domain to sibling -link servers"
philosophy already in AGENTS.md.

## 3. Scope (decided)

In scope (v0.4.0):

1. **G1** — add a uniprot-link hint to `see_also` (gene-keyed, like the
   genereviews/gtex hints), gated by `response_mode` exactly like the others.
2. **G2** — surface the resolver's molecular consequence as a top-level
   `molecular_consequence` field (when the input was HGVS/rsID and VEP returned
   one) and fold a short clause into the combined + single-model headline.
3. Capabilities/glossary updates documenting both; version bump to v0.4.0.

Out of scope: protein→genomic coordinate bridging (the consumer supplies
HGVS/rsID/coordinate; we delegate resolution to VEP), adding pubtator/other
servers to see_also (one well-chosen addition avoids re-inflating the per-call
`_meta` tax the original eval flagged), any new upstream call (G2 reuses the
consequence VEP already returned during resolution — zero added latency).

## 4. G1 — uniprot-link in consequence-aware see_also

**Where:** `spliceailookup_link/mcp/tools/_common.py` — `_see_also_full`.

**Change:** when `gene` is known, append a fourth hint:
```python
{
    "server": "uniprot-link",
    "hint": f"protein domains, features, and disease variants for {gene}",
    "example": {
        "tool": "find_proteins",
        "arguments": {"gene": gene, "organism_taxon": 9606, "reviewed": True},
    },
}
```
Ordering: place it after genereviews (gene-disease) and before/after gtex — group
the three gene-keyed sibling hints together. The `response_mode` gating in
`see_also_for` is unchanged: `minimal` → omitted; `compact` → `{server, hint}`
only; `full` → with `example`. So the added token cost in the default compact
mode is a single `{server, hint}` pair.

**Rationale:** uniprot-link is a first-class sibling in this ecosystem; for any
coding variant the protein-domain/feature context is the natural cross-domain
follow-up, and the self-test workflow literally started there. This makes the
loop bidirectional (uniprot → splice → uniprot) instead of a dead-end.

**Why gene-keyed, not consequence-gated:** protein context is relevant for any
variant overlapping a gene, and gating on the molecular consequence would add a
branch and a dependency on a field that is absent for coordinate inputs. Keeping
it parallel to the genereviews/gtex hints (present whenever `gene` is set) is
simpler and equally correct. The hint text is gene-level, not residue-level, so
it is accurate regardless of the specific consequence.

## 5. G2 — surface the molecular consequence

**Where:** `spliceailookup_link/mcp/tools/_predict.py` (combined),
`spliceailookup_link/mcp/tools/spliceai.py` / `pangolin.py` (single-model),
`spliceailookup_link/mcp/tools/_predict_shape.py` (headline).

**Data already available:** `prepare_variant` returns `PreparedVariant.consequence`
(the VEP `most_severe_consequence`) for HGVS/rsID inputs; `predict_one` already
carries it as `prepared.consequence` and surfaces `_meta.resolved_consequence`.
The single-model tools have `prepared.resolution`. **No new upstream call.**

**Change:**

1. **Top-level field.** When the resolver produced a molecular consequence
   (input was HGVS/rsID), add `result["molecular_consequence"] = <value>` to the
   combined result and the single-model results. Keep `_meta.resolved_consequence`
   for backward compatibility (it stays). The new field is the documented,
   top-level, decision-relevant home; it is distinct from the existing top-level
   `consequence` object (which is the SAI-10k *splice* aberration prediction).
   Omitted entirely for coordinate inputs (no VEP consequence) — additive, never
   null-padded.
2. **Headline clause.** `combined_headline` gains an optional trailing clause
   `; <molecular consequence, humanized>` when present, e.g.
   `... models agree (both low/none); missense variant.`. Humanize by replacing
   `_` with a space. The clause is appended after the splice/aberration tail so
   the splice answer still reads first (the tool's primary job). Single-model
   tools (`predict_spliceai`/`predict_pangolin`) get the top-level
   `molecular_consequence` field but **not** a headline clause — their headlines
   are built inside `shaping.py` (near the LOC budget) and the splice-only framing
   is appropriate for a single-model question; the combined tool is the
   "what does this variant do?" answer where the clause adds the most value.

**Naming:** `molecular_consequence` (VEP, e.g. `missense_variant`) vs the existing
top-level `consequence` (SAI-10k splice aberration). The glossary must state the
distinction explicitly so an agent never conflates them.

**Combined-tool plumbing:** `combined_headline` gains a
`molecular_consequence: str | None = None` parameter (keyword, defaulted, so the
F6 tests that call it without the arg keep passing). `predict_one` passes
`prepared.consequence`. The `minimal_combined` projection includes
`molecular_consequence` only if present (it is a single short string — cheap, and
genuinely decision-relevant even at the headline tier). Single-model tools pass
`prepared.resolution`-derived consequence to their headline builders the same way.

## 6. Module map (600-LOC budget)

All edits are small and additive:

- `mcp/tools/_common.py` (~138 → ~150) — G1 uniprot hint.
- `mcp/tools/_predict.py` — G2 top-level field + pass consequence to headline.
- `mcp/tools/_predict_shape.py` (~102) — G2 headline param + minimal field.
- `mcp/tools/spliceai.py` / `pangolin.py` — G2 top-level field + headline clause.
- `mcp/shaping.py` (~405) — single-model headline clause param (small; re-check
  `make lint-loc` stays < 600; if it would cross, the headline clause for
  single-model can be applied in the tool layer instead of shaping).
- `mcp/resources.py` — capabilities: document the uniprot delegation in
  limitations/see_also notes and `molecular_consequence` in the glossary.
- `__init__.py` / `pyproject.toml` — version → 0.4.0.

## 7. Testing (respx-mocked, deterministic)

- **G1:** `see_also` for a result with a known gene includes a `uniprot-link`
  hint; `compact` collapses it to `{server, hint}`; `full` keeps the `find_proteins`
  example; `minimal` omits all see_also (unchanged). Assert the hint count is now
  4 (gnomad + genereviews + gtex + uniprot) when gene is known, 1 (gnomad only)
  when gene is None.
- **G2:** an HGVS/rsID-resolved prediction (StubService.resolve returns a
  `consequence`) has top-level `molecular_consequence == "missense_variant"` and a
  headline containing "missense variant"; a coordinate input has NO
  `molecular_consequence` key and no molecular clause in the headline. Combined +
  single-model both covered. `minimal` combined includes `molecular_consequence`
  when present.
- **Regression:** the existing F6 `combined_headline` unit tests (which call it
  without `molecular_consequence`) still pass (defaulted param).
- **Capabilities:** glossary mentions `molecular_consequence` and `uniprot-link`;
  `capabilities_version` changes from the v0.3.0 value and is stable across calls.
- `make ci-local` green, coverage ≥ 80%, every module < 600 LOC.

## 8. Acceptance

- `make ci-local` green; all G1/G2 tests pass.
- `predict_splicing("<HGVS missense>")` returns top-level `molecular_consequence`,
  a headline that states both the splice verdict and the variant type, and a
  `see_also` that includes a ready-to-call uniprot-link `find_proteins` hint for
  the gene.
- Capabilities documents uniprot-link as the protein-context delegate and the
  `molecular_consequence` vs `consequence` distinction.
- Re-run the self-test workflow mentally/against a rebuilt server: the
  PNKP/uniprot loop now closes (splice answer hands back to protein context).

*Research use only; not for clinical decision support. Splice predictions are
computational and must be interpreted alongside orthogonal evidence.*
