# SpliceAILookup-Link MCP — UX → >9.5/10 Design

**Date:** 2026-06-12
**Author:** MCP engineering (LLM-consumer-facing UX)
**Driver:** `docs/mcp-ux-assessment.md` (LLM-consumer assessment, overall 8.5/10)
**Target version:** v0.8.0 (intentional, documented breaking change to the
`predict_splicing` sub-block shape; new `capabilities_version`)

---

## 1. Purpose

The newest consumer assessment (`docs/mcp-ux-assessment.md`) scores the server
8.5/10 overall. The contracts (errors 9.5, observability 9, discoverability 9,
chaining 9, safety 9) are already excellent. The two levers holding the score
down are:

- **Token efficiency (7)** — the default `predict_splicing` payload echoes the
  request parameters three times (envelope + `spliceai{}` + `pangolin{}`) and
  ships three overlapping headlines; `see_also` adds four cross-server entries
  on every predict.
- **Speed (8)** — the most common negative (`not_found`) costs a full ~20s cold
  upstream round-trip; the local REF pre-flight already proves a cheaper path
  exists.

Plus four small correctness/consistency/polish findings (D1–D3, D5) that, once
closed, leave the evaluator nothing left to dock.

This design closes **every genuinely-open item** the assessment raised. A prior
analysis confirmed (against live code) that the recently-merged `D1`–`D5`
commits belong to an *earlier* assessment and a different numbering; only the
predict-side of UX-D1 overlaps. Everything below is open.

### Grounding in current best practice

Research against primary sources (Anthropic engineering "Writing tools for
agents" 2025-09, "Code execution with MCP" 2025-11, "Advanced tool use"
2025-11; modelcontextprotocol.io 2025-06-18 / 2025-11-25) yields the principles
this design applies:

- **Return only high-signal, decision-relevant data; stop echoing the request.**
  Input params are already in the model's context from the call it just made.
- **Numbers as numbers, validated by an output schema** — strings-as-numbers is
  an anti-pattern (D3).
- **Lean by default, widen on demand** — `minimal | compact | full` is the
  endorsed `response_format` superset; the default should be the lean tier.
- **Errors should guide recovery and be honest** — a tool must not silently hand
  back a wrong answer (D1).
- **Tasks for 30s+ calls** — already supported (`execution.taskSupport=optional`).

## 2. Goals / Non-goals

**Goals**

- G1. Cut the default `predict_splicing` payload by removing the triplicated
  request echo and the duplicate per-model headlines, without losing any
  decision-relevant field. (Token efficiency → ~9.5)
- G2. Gate `see_also` independently of `next_commands`. (Token efficiency,
  chaining control)
- G3. `resolve_variant` never silently returns a confident `variant_id` for a
  wrong-REF coordinate; the schema docstring matches reality. (Correctness, D1)
- G4. Fast-fail `not_found` in <0.5s on the coordinate path. (Speed → ~9, D4)
- G5. Fix contig classification (D2), numeric `all_non_zero_scores` (D3), and the
  band-`none` headline (D5).
- G6. Keep standalone `predict_spliceai` / `predict_pangolin` outputs intact —
  they are the only place the request context lives for a single-model call.
- G7. `make ci-local` green; every change unit-tested (respx-mocked, no live
  upstream in default CI); 600-LOC/module budget preserved.

**Non-goals**

- N1. No new tools; no tool renames. (Schema *shape* of `predict_splicing`
  sub-blocks changes; tool names/semantics do not.)
- N2. No bundled local gene-model data file. D4 reuses the Ensembl pre-flight
  round-trip via an overlap query — no large static dataset to ship or keep
  fresh.
- N3. No `warm_ttl_remaining_s`. The assessment itself retired this (the
  `served_warm` boolean + the `rate_budget` error-path signal are deemed
  sufficient, and upstream Cloud Run warmth decay is not observable to us). A
  one-line capabilities note will state this explicitly so it is not read as an
  oversight.
- N4. No change to the upstream-error contract (HTTP 200 + `error` body) or to
  timeouts/concurrency caps.

## 4. Detailed design

### C1 — De-triplicate request params (combined only)

`shape_spliceai` / `shape_pangolin` legitimately emit
`variant_id / genome_build / gene_set / max_distance / mask` because the
standalone single-model tools (`predict_spliceai`, `predict_pangolin`) have no
other place to carry that context. In the *combined* path those five fields are
already on the envelope (`_predict.py` lines ~144–150), so the copies inside
`result["spliceai"]` and `result["pangolin"]` are pure redundancy (3× each).

**Approach.** Add a small private helper in `_predict.py`,
`_dedup_subblock(block)`, that pops the five envelope-owned keys from a shaped
sub-block. Call it on each present sub-block after shaping. Do **not** change
`shaping.py`’s emission — standalone tools must keep the fields.

Keys popped from `spliceai{}` / `pangolin{}` in combined:
`variant_id, genome_build, gene_set, max_distance, mask`.
Kept: `model, max_delta_score, transcripts, interpretation.band` (+ `headline`
in full only, see C2).

### C2 — Drop duplicate per-model headlines in combined compact

In combined `compact`, the per-model `spliceai.headline` / `pangolin.headline`
overlap the combined top-level `headline`. Pop them in `compact`; **keep** them
in `full` (a power user inspecting raw structure may want the per-model line).
`minimal` already projects via `minimal_combined` and is unaffected.

Implemented in the same `_predict.py` pass as C1, guarded by
`response_mode == "compact"`.

### C3 — Trim redundant per-transcript `max_delta_score` (single transcript)

When a sub-block has exactly one transcript, `transcripts[0].max_delta_score`
equals the model-level `max_delta_score`. Drop the per-transcript copy in that
case (compact + full of the combined path). With ≥2 transcripts (`transcripts=
all`) the per-transcript values differ and are kept. This is a minor trim folded
into the C1/C2 dedup pass; it also applies to standalone single-model tools
(safe: the model-level field is retained).

### C4 — Split `see_also` from `next_commands`

Today `include_hints=false` drops both `next_commands` and `see_also`. Add a
second boolean `include_see_also` (default `true`) to `predict_splicing`,
`predict_spliceai`, `predict_pangolin`. Semantics:

- `next_commands` emitted when `include_hints=true` (unchanged).
- `see_also` emitted when `include_hints=true` **and** `include_see_also=true`.

So a warm client can keep the hot chaining path (`next_commands`) and drop the
four cross-server `see_also` entries with `include_see_also=false`. `minimal`
continues to omit `see_also` regardless (existing behavior). Batch passes
`include_see_also=false` by default for per-item leanness (documented).

`_common.see_also_for` is unchanged; only the call sites gate it.

### C5 — resolve_variant REF validation (soft warning, default on)

**Behavior.** For a coordinate input, after contig/ambiguous handling,
`resolve_variant` calls a new non-raising variant of the REF pre-check and:

- on **match / inconclusive (Ensembl down) / non-ACGT REF**: sets
  `ref_validated: true` (match) or omits the flag (inconclusive — never claim
  validation we did not perform). Returns the normalized `variant_id` as today.
- on **mismatch**: sets `ref_validated: false`, adds
  `ref_warning: "REF '<typed>' does not match the <build> reference base
  '<actual>' at <chrom>:<pos>; predict_* will reject this as ref_mismatch.
  Re-check the allele or genome_build."`, and — when the typed REF matches the
  **other** build — appends the other-build hint already produced by
  `_diagnose._build_ref_mismatch`. Still returns `variant_id` (resolve remains a
  normalizer), but the answer is now honest.

**Refactor.** Extract the comparison core of `preflight_ref_mismatch` into a
shared helper returning a structured verdict instead of raising:

```
async def check_ref(service, *, variant_id, requested_build) -> RefCheck
# RefCheck: status in {match, mismatch, inconclusive}, requested_base,
#           observed_ref, other_build (optional)
```

`preflight_ref_mismatch` (predict_* path) becomes a thin wrapper that raises
`RefMismatchError` on `mismatch`; `resolve_variant` consumes the same `RefCheck`
to build the soft warning. Single source of truth, no duplicated Ensembl logic.

**Param.** New `check_ref: bool = True` on `resolve_variant` to skip the Ensembl
round-trip for latency-sensitive callers (the warning is then absent and
`ref_validated` is omitted).

**Docstring.** Replace the line "Coordinate inputs are normalized, not
validated: a wrong REF allele passes resolution and only fails at prediction
time." with: "Coordinate inputs are normalized; by default the REF base is also
checked against the requested build (one Ensembl lookup) and a `ref_warning` is
returned on mismatch (set `check_ref=false` to skip). HGVS/rsIDs are resolved
and validated via Ensembl VEP."

**Output schema.** Add `ref_validated: {type: [boolean, null]}` and
`ref_warning: {type: [string, null]}` to `_OUTPUT_SCHEMA`.

### C6 — Contig classification (D2)

`variant.py` currently lets a well-formed coordinate with chrom `99` fall
through to HGVS/rsID parsing → `VariantParseError` → `invalid_input`, while
`chrM` reaches `unsupported_contig`. The capabilities doc defines
`unsupported_contig` as "MT or non-standard", so a well-formed but unknown
contig is non-standard and belongs there.

**Approach.** In the coordinate parse path, when the input matches the
coordinate delimiter structure (chrom / integer pos / ACGT-ish ref / ACGT-ish
alt) **and** the chrom token is a pure integer outside 1–22 (e.g. `0`, `23`,
`99`, `100`), raise `UnsupportedContigError` with a reason like "Contig 'chr99'
is not a standard nuclear chromosome (1-22, X, Y). SpliceAI scores nuclear
chromosomes only." Genuinely malformed input (non-numeric junk, wrong field
count) continues to `invalid_input`. Non-integer odd contigs that are not
1–22/X/Y/M/MT but are *structurally* a coordinate (rare) also route to
`unsupported_contig`. Keep the existing `unsupported_contig_reason` for MT/M.

This is a localized change in the coordinate normalizer; add a precise predicate
so we never reclassify true `invalid_input` cases.

### C7 — Numeric `all_non_zero_scores` (D3)

In `shape_pangolin` (`shaping.py` ~441–446), the `all_non_zero_scores.scores`
dict is emitted with upstream string values (`"0.92"`). Add a `_coerce_floats`
pass that maps numeric-looking strings to floats (leaving non-numeric values
untouched). Apply to the `scores` mapping before emitting. Keeps every other
score field's float discipline consistent.

### C8 — Fast-fail `not_found` via Ensembl overlap (D4)

**New client method.** `EnsemblVepClient.overlapping_transcripts(chrom, pos,
build, window) -> int | None` querying
`/overlap/region/human/{c}:{start}..{end}?feature=transcript` (build-specific
host), where `start = max(1, pos - window)`, `end = pos + window`. Returns the
transcript count, or `None` on any upstream fault / unexpected shape (treated as
inconclusive, exactly like `reference_base`). `window = max_distance` (the
scan window SpliceAI uses); a variant with no transcript within `max_distance`
cannot be scored by either gene_set.

**Service.** Add `SpliceService.overlapping_transcripts(...)` delegating to the
client (cache-backed like the other Ensembl calls).

**Pre-flight.** In `prepare_variant` (coordinate branch), after the REF check
passes, when `settings.PREFLIGHT_OVERLAP_CHECK_ENABLED` and `cross_build_check`:
call the overlap check; if it returns a conclusive `0`, raise
`DataNotFoundError` (→ `not_found`) immediately. Any `None`/`>0` falls through
to real scoring. This guarantees the pre-check can only *confirm* a not_found,
never invent one.

**Config.** New `PREFLIGHT_OVERLAP_CHECK_ENABLED: bool = True` in `config.py`,
documented alongside `PREFLIGHT_REF_CHECK_ENABLED`.

**Ordering / cost.** On the coordinate path the pre-flight now makes up to two
Ensembl calls (REF region + transcript overlap), both fast (~200–400ms),
cache-backed, and independent of the rate-limited scoring backends. The win:
the common cold `not_found` drops from ~20s to <0.5s. HGVS/rsID inputs skip the
overlap check (VEP already proved a transcript context).

**Safety.** Conservative window + fall-through-on-inconclusive means a false
fast-fail is only possible if Ensembl authoritatively reports zero transcripts
within `max_distance` while GENCODE has one — vanishingly rare at the
gene-body/flank scale, and the failure mode is a `not_found` the real call would
very likely also return. A capabilities note documents the heuristic.

### C9 — Band-`none` headline (D5)

In `spliceai_headline` / `pangolin_headline`, when the top score's band is
`none` (max Δ == 0 or no non-zero delta), return
`"<Model> (<build>): <gene> — no predicted splicing impact (max Δ=0.00)."`
instead of `"... none acceptor gain (Δ=0.00 at +0 bp)."`. The combined headline
already prints explicit scores and needs no change.

### C10 — Discovery, docs, versioning

- `mcp/resources.py` capabilities doc: document the leaner combined shape
  (request params on the envelope only; per-model headlines in `full` only), the
  `include_see_also` flag, `resolve_variant`'s `ref_validated`/`ref_warning` +
  `check_ref`, the contig rule (D2), the overlap pre-check (D4), and the
  explicit "no `warm_ttl_remaining_s`; use `served_warm` + `rate_budget`" note.
  `capabilities_version` is a content hash and bumps automatically.
- `docs/API.md`: update the `predict_splicing` example payload and the
  `resolve_variant` contract.
- `pyproject.toml` → `0.8.0`; `uv.lock` synced.

`integration`-marked.

  them; single-transcript sub-block lacks per-transcript `max_delta_score`;

  false` → no Ensembl call (assert via mock call count) and no flag.
- **C6:** `chr99`, `chr0`, `chr23` → `unsupported_contig`; `chrM` still

- **C8:** zero-overlap → `not_found` with no scoring call (assert mock); ≥1

- **C10:** capabilities doc mentions the new fields; `capabilities_version`

## 7. Migration / breaking-change handling

- Bump to **0.8.0**. The only breaking shape change is the *combined*
  `predict_splicing` sub-blocks losing the five echoed request keys + (in
  compact) the per-model headlines. The envelope already carries all five, so a
  consumer reading them at the envelope is unaffected; only a consumer reading
  them from inside `spliceai{}`/`pangolin{}` must move up one level.
- The new `capabilities_version` content hash signals warm clients to
  re-discover; the capabilities doc spells out the moved fields.
- Additive, non-breaking: `include_see_also`, `check_ref`, `ref_validated`,
  `ref_warning`, the overlap pre-check, the contig reclassification (a stricter
  but more correct error code), the float coercion, the headline copy.

## 8. Risks & mitigations

- **R1 — false fast-fail (D4).** Mitigated by conservative `max_distance` window
  + fall-through on inconclusive; the pre-check can only confirm, never invent, a
  not_found. Setting-gated for instant rollback.
- **R2 — added resolve latency (D1).** One Ensembl call on coordinate resolves;
  `check_ref=false` opt-out; cache-backed. Acceptable for correctness.
- **R3 — breaking sub-block shape.** Explicitly authorized; documented; version
  + capabilities hash bump; standalone tools untouched.
- **R4 — LOC budget.** Tracked per-file above; split helpers if any file nears
  500. `make lint-loc` enforces.

## 9. Success criteria

- S1. `make ci-local` green (format, lint, lint-loc, mypy, tests).
- S2. Combined `predict_splicing` compact payload no longer contains the five
  request keys inside `spliceai{}`/`pangolin{}` nor per-model headlines;
  measured field/byte reduction recorded.
- S3. `resolve_variant` on a wrong-REF coordinate returns `ref_validated:false` +
  `ref_warning` and never a silent confident-only success.
- S4. A coordinate with no overlapping transcript returns `not_found` without a

- S5. `chr99` → `unsupported_contig`; Pangolin `all_non_zero_scores` numeric;
  band-`none` headline reads naturally; `see_also` independently gateable.
- S6. Capabilities + `API.md` updated; version 0.8.0; every open item in
  `docs/mcp-ux-assessment.md` Part 2 (#1–6) and Part 3 (D1–D5, prioritized

Net intended effect: Token efficiency 7→~9.5, Speed 8→~9, all nits closed →
overall **>9.5/10**.
