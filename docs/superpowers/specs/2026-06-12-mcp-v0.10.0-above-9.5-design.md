# Design: spliceailookup-link v0.10.0 — closing the v0.9.0 assessment to >9.5/10

**Date:** 2026-06-12
**Author:** Claude (Opus 4.8), acting as MCP engineer
**Input:** `docs/mcp-assessment-v0.9.0-2026-06-12.md` (scored 9/10)
**Target:** address every concrete recommendation in that assessment, grounded in
current MCP best practices, to clear >9.5/10.

---

## 1. Context and the one surprise

The v0.9.0 assessment is a black-box, LLM-consumer + senior-tester report. It
scored the server 9/10 and produced a tight, prioritized punch-list. Before
designing, I read the code for each recommendation and verified one claim
empirically:

> **Surprise:** the assessment's #1 recommendation — "MCP tool annotations are
> absent from all 7 schemas" — is **already implemented and correct in code**.
> All 7 tools carry `annotations=READ_ONLY_OPEN_WORLD`
> (`spliceailookup_link/mcp/annotations.py`), present since the initial commit,
> and FastMCP 3.4.2 **does** serialize them into `tools/list`. Verified by
> introspection: every tool emits
> `{readOnlyHint:true, destructiveHint:false, idempotentHint:true, openWorldHint:true}`.

So the tester's client (Fable 5) simply did not surface annotations to the
evaluating model. That reframes #1 from "add annotations" (a no-op) to
**"lock the behavior in with a regression test, and make read-only/idempotent
status discoverable through the capabilities channel the LLM always reads"** —
so even a client that hides `tools/list` annotations conveys the property.

Two other current-best-practice checks also came back already-satisfied:

- **`structuredContent`** (June-2025 spec) is already emitted — FastMCP returns
  the dict result as `structured_content`. No work needed.
- **`outputSchema`** is intentionally *not* adopted: these payloads are
  polymorphic by `response_mode` (minimal/compact/full) and by partial-failure
  shape. A rigid output schema would fight the deliberate shaping and add churn
  for negative value. Explicitly out of scope.

Everything else in the assessment is real. The plan below covers **all** of it.

## 2. Goals / non-goals

**Goals** — implement every concrete recommendation from Parts 1 and 2:

| # | Workstream | Assessment source |
|---|---|---|
| W1 | Lock in tool annotations + surface read-only status in capabilities | P1.1, P2.3 |
| W2 | Dedup `predict_splicing_batch` by resolved `variant_id` | P2 Finding 1, Rec 2 |
| W3 | Emit model/build **provenance** in prediction payloads | P1.2, P2 Finding 2, Rec 3 |
| W4 | On `not_found`, return distance to the nearest annotated transcript | P2 Finding 5 |
| W5 | Clarify `basic` gene-set scope (includes non-coding GENCODE) | P2 Finding 4, Rec 4 |
| W6 | Trim the hot-path `_meta` (drop per-call `capabilities_version`) | P1.3, Token-efficiency 8 |
| W7 | `warmup`: cover both masks in one shot + stay-warm estimate | P1.4 |
| W8 | Accept an optional client-supplied `correlation_id`, echoed in `_meta` | P1.5 |

**Non-goals:** new tools; renaming/removing tools or response keys; breaking
schema changes; `outputSchema`; raising `MAX_CONCURRENCY` or shortening
timeouts; any live-upstream behavior change. Backwards compatibility is
preserved — every change is additive or a documented token trim.

## 3. Design per workstream

Each is independently testable (respx-mocked / `StubService`), kept within the
600-LOC/module budget, and lands as an atomic commit with tests.

### W1 — Annotations: lock in + surface (cheap, high discoverability)
- **Regression test** asserting all 7 tools serialize
  `readOnlyHint=true, idempotentHint=true, openWorldHint=true,
  destructiveHint=false` via `list_tools()` → `to_mcp_tool().annotations`.
  This locks the property the tester couldn't observe.
- **Capabilities surfacing:** add a `tool_safety` block to the capabilities doc:
  `{all_tools_read_only: true, idempotent: true, open_world: true,
  auto_approve_safe: true, note: "..."}`. An LLM that never sees `tools/list`
  annotations still learns the tools are safe to auto-call from the doc it
  always reads. One line also added to `_INSTRUCTIONS`.
- No change to the annotation values themselves (already correct).

### W2 — Batch dedup by resolved `variant_id` (the marquee functional win)
The scarce resource is the rate-limited, 10–40s upstream. Today a batch that
contains the same variant twice (e.g. `16-2317763-T-A` and its HGVS
`NM_001089.3(ABCA3):c.875A>T`) scores it twice.
- New pipeline in a dedicated helper module `mcp/tools/_batch_dedup.py`
  (keeps `_batch_runner.py` under budget): **resolve all inputs → group by
  canonical `variant_id` → score each unique variant once → re-expand results
  to every original input position.** Resolution reuses `prepare_variant`/
  `service.resolve`; an input that fails to resolve (invalid/ambiguous/unsupported)
  stays a per-item error at its own position and is never grouped.
- Each original position keeps its own `request_id` and per-item `_meta`. The
  re-expanded copies are marked so the saving is observable:
  `_meta.deduped = {unique: N, duplicates: M}` and a per-item
  `cache: "deduped"` (or `served_from: "<canonical_id>"`) on the copies that
  reused a sibling's score.
- `summary` gains `unique_variants` and `upstream_calls_saved`.
- Ordering, retry, partial-failure isolation, and the existing envelope shape
  are preserved.
- **Test:** a batch with a coordinate + its HGVS asserts `StubService.score_calls`
  drops from 4 (2 models × 2 inputs) to 2 (2 models × 1 unique), both positions
  still carry a full result, and `upstream_calls_saved == 2`.

### W3 — Provenance in prediction payloads (reproducibility / citation)
Authoritative backend versions (sourced from `broadinstitute/SpliceAI-lookup`):
GENCODE **v44 basic** (GRCh38) / **v44lift37 basic** (GRCh37); Broad-modified
SpliceAI (MNV-aware) and refactored Pangolin; SAI-10k consequence calculator;
Ensembl VEP REST resolver.
- New small module `mcp/provenance.py` exposing `prediction_provenance(build)`:
  ```
  {
    "spliceai": "SpliceAI (Illumina; Jaganathan 2019) via Broad SpliceAI Lookup",
    "pangolin": "Pangolin (Zeng & Li 2022) via Broad SpliceAI Lookup",
    "consequence": "SpliceAI-10k consequence calculator",
    "transcript_annotation": "GENCODE v44 basic (GRCh38; v44lift37 for GRCh37)",
    "mane": "MANE Select default for transcripts='mane'",
    "resolver": "Ensembl VEP REST",
    "note": "Reflects the documented Broad backend configuration; the upstream
             does not assert versions per call."
  }
  ```
  The GENCODE version is a `settings` constant (`GENCODE_VERSION`,
  env-overridable) so an operator can bump it without a code change.
- Emitted as a top-level `provenance` block in `predict_spliceai`,
  `predict_pangolin`, `predict_splicing` (compact + full); **omitted in
  `minimal`** for token economy. For `predict_splicing_batch`, emitted **once**
  on the envelope (not per item).
- It is reproducibility *data*, not a chaining hint, so it is **not** gated by
  `include_hints`.
- Capabilities `data_sources` is upgraded from bare names to the same versioned
  detail (single source of truth via the provenance module).

### W4 — Nearest-transcript distance on `not_found`
- `EnsemblVepClient.nearest_transcript(chrom, pos, build, max_window=100_000)`:
  one `overlap/region` call over a capped window, returns the closest transcript
  `{distance_nt, gene, transcript_id}` (0 if the position is inside a transcript)
  or `None` (no transcript within the cap / any fault — never invents data).
- `SpliceService.nearest_transcript(...)` cached wrapper (mirrors
  `overlapping_transcripts`).
- The genuine-`not_found` paths (`preflight_no_overlap` and
  `diagnose_coordinate_failure`) attach the lookup to the raised
  `DataNotFoundError`; the error layer surfaces it in the `not_found` envelope as
  `nearest_transcript`, and the prose `recovery` mechanically advises widening
  `max_distance` when `distance_nt <= 10000` (else "nearest transcript is N bp
  away — beyond the 10 kb model window; this variant is intergenic for splicing").
- Best-effort: any Ensembl fault degrades to today's behavior (field omitted).

### W5 — Clarify `basic` gene-set scope (docs only)
- Capabilities `parameters.gene_set` and the `reference` resource state that
  `basic` = **GENCODE v44 _basic_** — includes non-coding genes (e.g. lncRNAs
  such as CCAT2 that have a basic/MANE transcript), not only protein-coding; a
  low-scoring lncRNA hit is a *real transcript*, not the absence of annotation.
- Matches the W3 provenance string; no behavior change.

### W6 — Trim the hot-path `_meta`
- Stop echoing `capabilities_version` in the per-call success `_meta`. It is
  static, already lives at the top of the capabilities doc, and the lean view
  already tells clients to read it from `get_server_capabilities`. Removing it
  from every prediction response is a pure token win on multi-call workflows.
- `served_warm`, `cache`, `rate_budget`, `timing`, `request_id` (all dynamic)
  stay. The `unsafe_for_clinical_use` disclaimer stays (always).
- Capabilities documents the change (warm clients get the hash from the doc).

### W7 — `warmup`: both masks + stay-warm estimate
- `mask` accepts `"both"` (in addition to `raw`/`masked`); when `both`, the
  service warms raw **and** masked per model in one call. Default stays `raw`
  (no increase to default upstream load — `both` is explicit opt-in).
- Response adds `stay_warm_estimate_s` (a conservative Cloud Run idle-decay
  estimate, config constant `WARMUP_STAY_WARM_ESTIMATE_SECONDS`, default ~900s)
  with an explicit "estimate, not a guarantee" caveat, and `coverage.mask`
  reflects what was warmed (`raw` | `masked` | `both`).

### W8 — Optional client-supplied `correlation_id`
- Optional `correlation_id: str | None` (Field `max_length=128`) on
  `predict_spliceai`, `predict_pangolin`, `predict_splicing`,
  `predict_splicing_batch`, `resolve_variant`.
- Threaded into `run_mcp_tool(..., correlation_id=...)`; `_stamp` echoes it as
  `_meta.correlation_id` on **success and error** so a multi-step workflow is
  traceable as one unit. For batch, echoed once on the envelope.
- Aligns with the "operational excellence: correlation IDs" best practice.

## 4. Cross-cutting concerns
- **File-size budget:** `_batch_dedup.py` and `provenance.py` are new small
  modules so no existing file approaches 600 LOC. `errors.py` (551) gains only a
  few lines in `_stamp` (W6/W8) — net neutral after dropping the
  `capabilities_version` block; will re-check `lint-loc`.
- **Versioning:** bump `__version__` and `pyproject` to **0.10.0**. The
  capabilities content hash (`capabilities_version`) will change — expected and
  documented (new `tool_safety`, richer `data_sources`, doc clarifications).
- **Testing:** TDD, respx/`StubService`, no live upstream in default CI. New
  tests live in `tests/unit/` (e.g. `test_v0_10_0.py` per theme). `make ci-local`
  (format, lint, lint-loc, mypy, tests) must be green before handoff.
- **Docs:** update `docs/mcp-assessment-v0.9.0-2026-06-12.md` companion or add a
  short `docs/mcp-resolution-v0.10.0-2026-06-12.md` mapping each recommendation →
  change → test, mirroring the existing resolution-doc pattern.

## 5. Score impact (why this clears >9.5)
Addressing the full punch-list lifts exactly the dimensions that were < 10:
Speed (W2 saves the rate-limited upstream), Token efficiency (W6), Observability
(W3, W8), Error handling (W4), Discoverability (W1, W3), Ergonomics (W7), Safety/
Docs (W5). No dimension is left with an open assessment finding.

## 6. Self-review
- No placeholders/TBDs. Each workstream names its files, shape, and test.
- Internal consistency: provenance strings are single-sourced (W3 module) and
  reused by W5 docs and capabilities `data_sources`.
- Scope: additive/back-compatible; the only removal (W6 `capabilities_version`
  from per-call `_meta`) is documented and the value remains discoverable.
- Ambiguity resolved by explicit defaults: provenance omitted in `minimal`;
  `warmup` default mask stays `raw`; `correlation_id` is opt-in; batch provenance
  once on the envelope.
